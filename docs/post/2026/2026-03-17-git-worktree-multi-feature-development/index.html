Git Worktree 완전 가이드: 핫픽스, 정리, 복구, 베어 레포 전략까지

기능 개발을 하다가 갑자기 운영 장애가 터지면 보통 두 가지 나쁜 선택지로 끌려갑니다. git stash 로 급히 덮어 두거나, 어수선한 상태 그대로 임시 커밋을 만들거나, 심하면 저장소를 통째로 다시 복제합니다.
이 영상이 보여 주는 핵심은 Git Worktree가 이런 컨텍스트 전환 비용을 구조적으로 줄여 준다 는 점입니다. 하나의 저장소 히스토리를 공유하면서도 브랜치별 작업 디렉터리를 물리적으로 분리해, 피처 개발과 핫픽스를 동시에 다룰 수 있다는 설명입니다. 근거 영상

다만 이 주제는 실전 팁과 Git 자체의 공식 동작을 구분해서 이해하는 편이 훨씬 안전합니다. 그래서 이 글은 영상 내용을 그대로 옮기지 않고, git-worktree 문서와 릴리스 노트를 함께 확인해 실전에서 바로 쓸 수 있는 규칙 과 관행으로 받아들여야 하는 조언 을 나눠서 정리합니다. 근거 영상

1) Git Worktree는 정확히 무엇을 분리하고 무엇을 공유할까

영상은 Worktree를 “도서관의 여러 책상” 비유로 설명합니다. 하나의 Git 저장소를 공유하면서도, 각 브랜치를 서로 다른 물리 폴더에 펼쳐 두고 오갈 수 있다는 설명인데, 이 비유는 공식 문서와도 잘 맞습니다. git-worktree 문서 기준으로 Git은 하나의 저장소에 여러 working tree를 둘 수 있고, 객체 데이터베이스와 대부분의 저장소 메타데이터는 공유하되 HEAD, index, 일부 관리 정보는 worktree별로 따로 유지합니다. 그래서 브랜치를 바꿀 때 기존 파일을 덮어쓰지 않고도 동시 작업이 가능합니다. 근거 영상

여기서 실무적으로 가장 중요한 포인트는 “프로젝트를 다시 복제하는 것” 과 “같은 저장소의 linked worktree를 추가하는 것” 이 완전히 다르다는 점입니다. Worktree는 Git 객체를 새로 다운로드하는 방식이 아니라 기존 저장소의 공용 데이터를 재사용하므로, 저장소 이력 자체를 복제하지 않습니다. 그래서 디스크 비용을 줄이면서도 브랜치별 작업 공간은 독립적으로 유지할 수 있습니다. 근거 영상

이 구조를 이해하면 흔한 오해도 같이 풀립니다. Worktree가 Git 데이터를 공유한다고 해서 node_modules, Python 가상환경, 빌드 산출물까지 자동으로 공유되는 것은 아닙니다. 공유되는 것은 Git 저장소 관점의 객체와 메타데이터이고, 실제 워킹 디렉터리 안의 의존성 폴더는 worktree마다 별도입니다. 그래서 새 worktree를 만들면 보통 그 디렉터리 안에서 다시 npm install 또는 필요한 환경 셋업을 해야 합니다. 이 부분은 영상의 실전 설명과 공식 문서의 저장소 구조 설명이 함께 뒷받침합니다. 근거 영상

2) 가장 먼저 익혀야 할 패턴: 피처 개발 중 긴급 핫픽스 넣기

영상의 초반 시나리오는 아주 현실적입니다. 로그인 기능을 개발하던 중 아직 커밋하지 않은 수정이 남아 있는데, 운영 서버 장애 때문에 즉시 핫픽스를 넣어야 하는 상황입니다. 여기서 Worktree의 핵심 가치는 “현재 더러운 작업 디렉터리를 건드리지 않고” 별도 폴더에서 메인 브랜치 기반 작업 공간을 여는 데 있습니다. 발표자는 git worktree add 로 새로운 폴더를 만들고 그쪽에서 핫픽스를 처리한 뒤, 다시 원래 폴더로 돌아오면 기존 수정 상태가 그대로 남아 있다는 점을 강조합니다. 근거 영상

이 흐름을 명령어 수준으로 정리하면 아래처럼 이해하면 됩니다. 공식 문서 기준으로 git worktree add <path> <commit-ish> 는 지정한 경로에 새 worktree를 만들고 원하는 브랜치나 커밋을 체크아웃합니다. 즉 “새 브랜치로 갈아타기” 가 아니라 “새 브랜치를 다른 폴더에서 펼치기” 에 가깝습니다.

# 기능 개발 중인 기존 저장소에서
git status

# 저장소 바깥쪽 형제 디렉터리에 핫픽스용 worktree 생성
git worktree add ../repo-hotfix main

# 새 worktree로 이동해서 핫픽스 진행
cd ../repo-hotfix
git switch -c hotfix/prod-issue
# 수정 -> 테스트 -> 커밋 -> 푸시

sequenceDiagram
  participant F as 기능 작업 폴더
  participant G as Git 저장소
  participant H as 핫픽스 worktree

  F->>F: 미커밋 변경 유지
  F->>G: git worktree add ../repo-hotfix main
  G-->>H: 새 작업 폴더 생성
  H->>H: hotfix 브랜치 생성 후 수정
  H->>G: 커밋/푸시
  H-->>F: 원래 작업 상태는 그대로 유지

여기서 영상이 강하게 권장하는 “프로젝트 폴더 바깥에 worktree를 만들라” 는 규칙은 실무적으로 매우 좋은 조언이지만, Git의 하드 제약은 아닙니다. 공식 문서는 worktree 경로를 특정 위치로 제한하지 않습니다. 다만 저장소 내부에 worktree 폴더를 만들면 원래 worktree에서 그 폴더가 일반 파일처럼 보이거나, 실수로 관리가 지저분해질 수 있어서 형제 디렉터리로 분리하는 관행 이 널리 권장됩니다. 즉 이 부분은 “공식 규칙” 보다는 “운영상 안전한 패턴” 으로 이해하는 편이 맞습니다. 근거 영상

또 하나 기억할 점은 같은 브랜치를 두 개의 linked worktree에서 동시에 체크아웃하는 문제입니다. Git은 기본적으로 이런 상황을 막습니다. 영상이 설명하듯 같은 브랜치를 여러 작업 폴더에서 동시에 수정하면 충돌과 혼란이 커지기 때문입니다. 공식 문서 기준으로 이런 경우 --detach 를 사용해 detached HEAD 상태의 임시 worktree를 만들 수 있고, 보다 강하게는 git checkout --ignore-other-worktrees 같은 우회도 존재하지만, 일반적인 실전 운영에서는 --detach 또는 새 브랜치 생성 쪽이 더 안전합니다. 근거 영상

3) Worktree를 잘 쓰려면 생성보다 정리와 복구를 더 정확히 알아야 한다

Worktree를 몇 번만 써 보면 생성보다 관리가 더 중요하다는 사실을 금방 체감하게 됩니다. 영상 2편 파트에서 발표자는 list, remove, prune, lock, repair 를 각각 어떤 상황에서 써야 하는지 구분합니다. 공식 문서도 같은 방향을 취합니다. list 는 현재 연결된 worktree 상태를 보는 명령이고, remove 는 worktree와 그 관리 파일을 정상 절차로 지우는 명령이며, prune 은 수동 삭제로 인해 남은 stale 관리 정보를 정리하는 용도입니다. 근거 영상

실무에서 특히 중요한 차이는 remove 와 폴더 직접 삭제의 차이입니다. 영상을 보면 발표자가 “그냥 rm -rf 하지 말라” 는 점을 강조하는데, 이유가 분명합니다. git worktree remove 는 dirty 상태를 검사하고 필요하면 중단해 주는 안전 장치가 있습니다. 반면 사용자가 Finder나 rm 으로 worktree 폴더를 먼저 지워 버리면 Git은 메타데이터만 남긴 채 그 worktree가 여전히 존재한다고 믿을 수 있습니다. 이때 쓰는 정리 명령이 git worktree prune 입니다. 근거 영상

  C --> F["안전하게 정리"]
  E --> G["유령 worktree 기록 제거"]

복구 파트는 영상에서 특히 유용한 부분입니다. 발표자는 “폴더를 지웠지만 아직 prune 하지 않았고, 그 안에 staged 변경이 남아 있는 경우” 와, 이미 prune 해서 메타데이터가 사라진 경우를 구분해야 한다고 설명합니다. 이 차이는 매우 중요합니다. prune 전에는 worktree 관리 정보가 살아 있어서 복구 여지가 있지만, prune 후에는 stale 메타데이터가 제거되므로 이전 상태로의 복원이 훨씬 어려워집니다. 즉 복구 가능성의 경계는 단순히 폴더 삭제 자체보다 관리 메타데이터가 아직 남아 있느냐 에 가깝습니다. 근거 영상

이와 연결해서 lock 과 repair 도 알아둘 가치가 큽니다. 공식 문서 기준으로 lock 은 이동식 저장장치나 일시적으로 마운트 해제될 수 있는 환경의 worktree가 prune 대상이 되지 않도록 보호하는 기능입니다. repair 는 수동 이동이나 관리 파일 손상으로 worktree 연결이 깨졌을 때 다시 연결 상태를 복원하는 데 쓰이며, Git 2.29 릴리스에서 추가되었습니다. 즉 영상 후반의 복구 시나리오는 단순 팁이 아니라 Git이 공식적으로 제공하는 관리 기능 위에 서 있는 설명입니다. 근거 영상

4) 고급 사용: 베어 레포 + 여러 worktree로 리뷰/핫픽스/기능 개발을 병렬화하기

영상 3편 파트는 한 단계 더 나아가 베어 레포 기반 구조를 소개합니다. 핵심 아이디어는 저장소의 “두뇌” 와 실제 코드를 담은 “몸” 을 분리하는 것입니다. 즉 루트에는 bare repository만 두고, 실제 개발은 main, staging, review-pr-42 같은 worktree 폴더에서만 이루어지게 만드는 방식입니다. Git 관점에서 bare repository는 워킹 트리가 없는 저장소이고, 여기에 worktree를 붙이면 여러 작업 공간을 보다 의도적으로 운영할 수 있습니다. 근거 영상

이 구조가 특히 잘 맞는 장면은 동료 PR 리뷰와 병렬 개발입니다. 영상은 원래 하던 기능 개발을 건드리지 않고 특정 PR을 fetch 해서 별도 review worktree로 열고, 거기서 테스트와 확인을 마친 뒤 바로 정리하는 패턴을 보여 줍니다. 실무적으로 보면 이 방식의 장점은 명확합니다. 첫째, 리뷰 때문에 현재 브랜치 상태를 stash 할 필요가 없습니다. 둘째, PR 검증 중 마음껏 파일을 망가뜨려도 원래 작업공간에는 영향이 없습니다. 셋째, 리뷰가 끝나면 worktree와 임시 브랜치를 함께 지워 작업 흔적을 짧게 유지할 수 있습니다. 근거 영상

다만 영상이 후반에 짚는 비용 이야기도 꼭 같이 봐야 합니다. Worktree는 저장소 복제 비용을 크게 줄여 주지만, 의존성 설치 시간까지 공짜로 만들지는 못합니다. 특히 프론트엔드나 대형 애플리케이션에서는 새 worktree마다 패키지 설치와 환경 설정이 다시 필요할 수 있고, 이것이 디스크보다 더 큰 비용이 될 수 있습니다. 발표자가 PNPM 같은 도구와 빠른 정리 습관을 함께 권하는 이유도 여기에 있습니다. 즉 Worktree는 “무조건 빠른 도구” 라기보다, 저장소 전환 비용을 줄이는 대신 환경 준비 비용은 여전히 관리해야 하는 도구 로 보는 편이 정확합니다. 근거 영상

마지막으로 상대 경로 관련 이야기도 짚고 넘어갈 필요가 있습니다. 영상은 Git 2.48 이상에서 relative path 지원이 더 안전해졌다고 설명하는데, 공식 문서 기준으로 worktree.useRelativePaths 설정과 --relative-paths 옵션은 현재 지원되는 기능입니다. 다만 이 설정은 상대 경로 기반 연결 정보를 사용하게 만들어 이동성에는 유리할 수 있지만, 오래된 Git 버전과의 호환성에는 영향을 줄 수 있습니다. 그래서 이 부분은 “무조건 켜야 하는 옵션” 이 아니라, worktree와 저장소를 함께 옮길 가능성이 큰 환경에서 검토할 만한 설정으로 이해하는 것이 좋습니다.

핵심 요약

  • Git Worktree는 하나의 저장소 히스토리와 객체를 공유하면서도 브랜치별 작업 디렉터리를 분리해, 피처 개발과 핫픽스를 동시에 진행할 수 있게 해 줍니다. 근거 영상
  • git worktree add 의 본질은 브랜치를 갈아타는 것이 아니라, 다른 경로에 새로운 작업 공간을 열어 현재 작업 상태를 건드리지 않는 데 있습니다. 근거 영상
  • 저장소 바깥 형제 디렉터리에 worktree를 만드는 것은 매우 좋은 관행이지만, Git 문서가 강제하는 규칙은 아닙니다. 이 차이를 구분해서 이해해야 합니다. 근거 영상
  • 정리 단계에서는 remove 와 prune 의 차이를 정확히 알아야 하고, lock 과 repair 는 이동식 환경이나 손상 복구 상황에서 매우 유용합니다. 근거 영상
  • 베어 레포 + 다중 worktree 패턴은 PR 리뷰, 핫픽스, 기능 개발을 병렬화하는 데 강력하지만, 각 worktree의 의존성 설치와 환경 준비 비용은 별도로 관리해야 합니다. 근거 영상

결론

이 영상이 잘 보여 준 것은 Git Worktree가 단순한 편의 명령어가 아니라, 컨텍스트 전환을 저장소 구조 차원에서 설계하는 도구 라는 점입니다. 기능 개발 중간에 핫픽스를 넣고, 동료 PR을 별도 공간에서 검토하고, 필요 없어진 작업 공간을 안전하게 정리하는 흐름까지 모두 한 저장소 안에서 관리할 수 있기 때문입니다. 근거 영상

실전에서 가장 좋은 출발점은 복잡한 베어 레포 전략부터 도입하는 것이 아니라, 먼저 지금 쓰는 프로젝트에서 형제 디렉터리에 hotfix용 worktree 하나를 만들어 보는 것입니다.
그 한 번의 경험만으로도 왜 많은 개발자가 stash 중심 흐름보다 Worktree 기반 병렬 작업을 더 선호하는지 바로 체감할 수 있습니다. 근거 영상
