OpenCode + Neovim + tmux 실전 입문: 터미널 중심 AI 코딩 워크플로우 구축 가이드

터미널 중심 개발 환경(Neovim + tmux)을 유지하면서 AI 코딩 에이전트를 붙이고 싶다면, OpenCode는 꽤 현실적인 선택지입니다. 이 글은 영상 데모를 기준으로 실제로 바로 따라 할 수 있게 핵심 흐름만 압축해 정리합니다.

Sources

  • https://www.youtube.com/watch?v=9FW43mb1vTM

왜 OpenCode 조합이 잘 맞는가

영상의 메시지는 명확합니다. “코드를 AI에게 통째로 맡기기"보다, 반복 작업과 탐색 속도를 높이는 보조 에이전트로 쓰는 방식이 생산성이 좋다는 점입니다.

flowchart LR
    A[Neovim 코드 작성] --> B[OpenCode 질의/변경 제안]
    B --> C[tmux에서 실행 결과 확인]
    C --> D[수정 반영 또는 되돌리기]
  • OpenCode는 프로젝트 전체 문맥을 읽어 답변/변경 제안을 수행
  • 사용자는 코드 제어권을 유지하고 필요한 부분만 위임
  • tmux와 함께 쓰면 편집, 실행, 에이전트 대화를 한 화면 구조로 운영 가능

설치와 로그인: 가장 짧은 시작 경로

영상에서는 Homebrew 설치와 opencode auth login 흐름을 보여줍니다. Anthropic(Claude Pro/Max) 경로를 추천하지만, API 키 기반 다른 제공자(OpenAI 등)도 선택 가능합니다.

sequenceDiagram
    participant U as User
    participant T as Terminal
    participant W as Browser
    U->>T: OpenCode 설치
    U->>T: opencode auth login
    T->>W: 인증 링크 제공
    U->>W: 승인 후 코드 복사
    U->>T: 코드 입력
    T-->>U: 로그인 완료

운영 포인트:

  1. 프로젝트 크기에 따라 요금제/모델 선택 비용이 크게 달라짐
  2. 팀 표준이 없다면 먼저 소형 프로젝트로 과금/응답 품질을 검증

tmux 배치 전략: Pane보다 Window 분리가 더 안정적

영상에서는 두 가지를 비교합니다.

  • 같은 창에서 pane 분할로 OpenCode/에디터/실행 로그를 동시 표시
  • OpenCode를 별도 tmux window(AI)로 분리해 전환
flowchart TD
    A[tmux Session] --> B[Window 0: editor + app]
    A --> C[Window 1: opencode]
    B --> D[빠른 코드 수정/실행]
    C --> E[질의/변경 요청]

실무에서는 window 분리 방식이 더 깔끔한 경우가 많습니다. 컨텍스트 전환이 명확하고, OpenCode 화면을 키우기 쉬워 응답 검토가 편합니다.

/init와 기본 에이전트(plan/build) 이해

프로젝트 루트에서 /init을 실행하면 AGENTS.md가 생성되고, OpenCode가 구조와 패턴을 더 잘 파악하게 됩니다.

flowchart LR
    A[/init 실행] --> B[프로젝트 분석]
    B --> C[AGENTS.md 생성]
    C --> D[문맥 기반 질의 정확도 향상]

영상 기준 기본 에이전트 역할:

  • plan: 분석/설명/변경안 제시(실제 수정 없음)
  • build: 실제 코드 변경 적용

권장 루프:

  1. plan으로 “무엇을 바꿀지” 먼저 확인
  2. 제안 검토 후 build로 적용
  3. 결과가 다르면 즉시 되돌리기

되돌리기와 공유: 협업 안정성의 최소 장치

영상 데모에 나온 핵심 안전 장치:

  • /undo, /redo로 빠른 롤백/재적용
  • /share로 세션 링크 공유, /unshare로 공유 해제
flowchart LR
    A[변경 적용] --> B{의도대로인가?}
    B -->|No| C[/undo]
    B -->|Yes| D[유지]
    D --> E[/share 필요 시]
    E --> F[/unshare로 회수]

주의:

  • 공유 링크에는 코드 맥락이 포함될 수 있으므로 민감 정보가 있는 세션은 공유 금지

opencode.json 커스터마이징: 팀 표준의 시작점

영상에서는 프로젝트 루트 opencode.json에서 모델, 테마, 에이전트, 키바인드를 조정하는 흐름을 보여줍니다.

핵심 운영 항목:

  1. 리더 키(prefix) 통일
  2. 팀 공통 keymap 표준화
  3. 프로젝트별 기본 에이전트 정책 명시

키바인드를 바꿨다면 OpenCode 재시작으로 반영 여부를 바로 확인하는 루프가 필요합니다.

커스텀 명령으로 반복 보일러플레이트 제거

영상의 대표 예시는 .opencode/command/component.md 커맨드로 Svelte 컴포넌트 생성 자동화입니다. 인자($ARGUMENTS)를 받아 원하는 템플릿 규칙(예: TypeScript, 빈 props 인터페이스)을 고정합니다.

flowchart TD
    A[사용자: /component 이름] --> B[command markdown 로드]
    B --> C[지정 규칙으로 파일 생성]
    C --> D[lib/components에 결과 반영]

이 패턴은 프레임워크와 무관하게 재사용 가능합니다.

  • API 핸들러 스캐폴드
  • 테스트 파일 초기화
  • 폴더/파일 네이밍 규칙 강제

바로 적용하는 10분 체크리스트

  1. 프로젝트 루트에서 OpenCode 실행 후 /init 완료
  2. tmux에 editor/app와 AI window 분리
  3. 변경 요청은 plan -> build 순서 고정
  4. /undo와 /redo 동작을 먼저 검증
  5. opencode.json에 최소 keymap/모델 정책 정의
  6. .opencode/command에 팀 반복 작업 1개부터 자동화
  7. /share는 민감 정보 점검 후 제한적으로 사용

결론

이 영상의 핵심은 “AI가 코드를 대신 짜준다"가 아니라, 터미널 워크플로우를 깨지 않고 AI를 작업선에 끼워 넣는 방법입니다. Neovim + tmux 사용자는 OpenCode를 별도 창으로 분리하고, plan/build/undo 루프와 커스텀 명령만 잘 잡아도 체감 생산성이 빠르게 올라갑니다.
